# MeteoSignal v1.5.0 — Spécification

Ce document constitue la référence officielle de conception de MeteoSignal v1.5.0. Il fixe le périmètre, les choix d'architecture et les critères de validation avant toute implémentation fonctionnelle.

## 1. Statut

- Version publique actuelle : **1.4.2**.
- La version **1.5.0** est en préparation.
- Aucune fonctionnalité v1.5.0 n'est encore publiée.
- MeteoSignal Android/TWA v1.4.2, `versionName 1.4.2` et `versionCode 2`, est acceptée sur la piste de test fermé Google Play.
- Le test physique de la version distribuée et l'accès à la production publique sont suivis séparément.

La version publique et le cache PWA restent en 1.4.2 pendant le développement. Leur passage en 1.5.0 intervient uniquement pendant la préparation de la publication.

## 2. Objectifs

- unifier la navigation et l'accès aux villes enregistrées ;
- transformer la carte météo principale en expérience immersive ;
- ajouter des fonds météo contextuels ;
- optimiser de manière mesurée le tableau de bord desktop ;
- conserver l'accessibilité, les performances, la PWA et la compatibilité TWA.

## 3. Hors périmètre

La v1.5.0 n'intègre pas :

- de second fournisseur météo ni de données Météo-France ;
- de radar ou de vigilance officielle ;
- de réglages d'unités ;
- de Mode Expert ;
- de widgets ou de notifications ;
- de compte utilisateur ;
- de paiement, d'abonnement ou de publicité ;
- de modification du projet Android.

## 4. Matrice météo vers scènes

| Codes Open-Meteo | Conditions | Scène | Règle complémentaire |
|---|---|---|---|
| `0–1` | Ciel dégagé à principalement dégagé | `clear-day` ou `clear-night` | Utiliser `current.is_day` pour distinguer le jour et la nuit. |
| `2–3` | Partiellement nuageux à couvert | `cloudy` | Conserver une lumière adaptée à l'état jour/nuit. |
| `45`, `48` | Brouillard et brouillard givrant | `fog` | Utiliser `cloudy` seulement comme repli secondaire. |
| `51–67`, `80–81` | Bruine, pluie et averses | `rain` | Assombrir le voile la nuit sans changer de catégorie. |
| `82`, `95`, `96`, `99` | Forte averse, orage et grêle | `storm` | Conserver les éclairs hors de la zone de texte. |
| `71–77`, `85–86` | Neige et averses de neige | `snow` | Adapter le voile afin de conserver le contraste. |
| Code absent ou inconnu | Condition indisponible | `default` | Afficher le fallback CSS bleu nuit et cyan. |

La classification existante des codes météo reste la référence. La couche de scène est une représentation visuelle dérivée et ne modifie pas le modèle métier.

## 5. Architecture graphique

L'architecture retenue est hybride :

- une seule image WebP contextuelle pour l'atmosphère active ;
- des overlays, dégradés, halos et voiles de contraste en CSS ;
- un voile stable derrière la ville, la température, la condition, les valeurs minimum, ressenti et maximum, la source et l'heure de mise à jour ;
- un fallback CSS bleu nuit et cyan utilisable sans image ;
- aucune ressource externe ;
- aucune vidéo, aucun WebGL et aucune animation permanente complexe ;
- aucune dépendance supplémentaire ;
- aucun style inline incompatible avec la CSP.

L'image reste décorative, avec une alternative vide, et ne porte aucune information indispensable.

## 6. Ressources prévues

Les ressources seront placées dans le répertoire `assets/backgrounds/hero/`, avec les noms suivants :

```text
hero-clear-day.webp
hero-clear-night.webp
hero-cloudy.webp
hero-rain.webp
hero-storm.webp
hero-snow.webp
hero-fog.webp
```

Ces fichiers ne sont pas créés pendant la phase de spécification.

Contraintes de production :

- format WebP ;
- cible de 120 à 160 Kio par fichier ;
- maximum absolu de 200 Kio ;
- dimensions de travail proches de 1600 × 900 pixels ;
- zones importantes compatibles avec les recadrages mobile, tablette, desktop et très grand écran ;
- aucun texte intégré aux images.

## 7. Chargement et cache

- Une seule scène active est chargée.
- L'ancienne scène reste visible jusqu'au chargement réussi de la suivante.
- Le remplacement est atomique afin d'éviter un écran vide ou un clignotement.
- Un jeton de requête, ou un mécanisme équivalent, interdit à une ancienne demande de remplacer une scène plus récente.
- Les sept scènes ne sont pas ajoutées au précache.
- Un cache dynamique limité aux scènes réellement consultées sera implémenté dans une phase ultérieure.
- Le cache est versionné et purgé avec les versions précédentes de la PWA.
- Le fallback CSS reste disponible hors ligne lorsqu'une scène n'est pas en cache.
- Les API Forecast, Air Quality et Geocoding restent hors Cache Storage.

## 8. Navigation unifiée

La navigation utilise un seul panneau DOM canonique pour les villes enregistrées :

- deux emplacements d'accueil peuvent exister dans la structure, mais la liste n'est jamais dupliquée ;
- le même nœud est déplacé selon le breakpoint ;
- la liste peut être repliée ;
- le bouton expose le compteur et les attributs `aria-expanded` et `aria-controls` ;
- la ville active conserve un état accessible explicite ;
- la sélection et la suppression restent disponibles ;
- après suppression, le focus passe au contrôle voisin ou au bouton favori si la liste devient vide ;
- l'état vide reste annoncé clairement ;
- les clés et le format `localStorage` existants sont conservés ;
- aucune migration et aucune suppression de favoris ne sont réalisées.

## 9. Comportement responsive

### Desktop, à partir de 1180 px

Le panneau canonique rejoint la sidebar. En mode réduit, seuls une indication accessible et le compteur restent visibles. L'ouverture restaure les informations complètes.

### Tablette, entre les seuils actuels

L'accès aux villes enregistrées devient un contrôle compact intégré à la navigation. Le panneau s'ouvre dans le flux et ne recouvre ni la recherche ni le contenu.

### Mobile

La bottom navigation conserve ses cinq entrées actuelles. Les villes enregistrées restent accessibles dans le flux sous une forme compacte et repliable. Les suggestions de recherche conservent toujours la priorité visuelle et tactile.

Le responsive est contrôlé de 360 à 1920 pixels, sans considérer le mobile comme une simple réduction du desktop.

## 10. Optimisation desktop

- réduire de manière mesurée la hauteur du hero ;
- mieux utiliser sa partie droite avec l'atmosphère contextuelle, sans ajouter d'information redondante ;
- conserver des dimensions confortables pour la température, la ville et les textes ;
- conserver les six indicateurs en grille 3 × 2 autour de 1280 px ;
- afficher les six indicateurs sur une ligne uniquement lorsque l'espace utile est suffisant ;
- étudier une colonne secondaire facultative au-delà de 1600 px ;
- ne pas comprimer excessivement les prévisions horaires, les sept jours, les alertes ou les données Soleil et Lune ;
- ne pas modifier les largeurs globales et breakpoints de structure sans justification mesurée.

## 11. Accessibilité

- contraste suffisant sur toutes les scènes ;
- voile stable derrière les textes ;
- image décorative avec `alt=""` ;
- focus visible et ordre de focus cohérent ;
- zones tactiles d'au moins 44 × 44 pixels ;
- `aria-expanded` et `aria-controls` pour le panneau repliable ;
- restauration du focus après suppression ;
- validation aux zooms 200 %, 300 % et 400 % ;
- reflow sans défilement horizontal de page ;
- transitions désactivées ou quasi instantanées avec `prefers-reduced-motion` ;
- parcours manuels avec NVDA et TalkBack ;
- fonctionnement complet sans image de scène.

## 12. Budget de performance

- une seule scène initiale ;
- 160 Kio maximum recommandé par scène ;
- 200 Kio maximum absolu ;
- 180 Kio maximum recommandé lors d'un changement de scène ;
- transfert initial cible inférieur à 1 Mio après les optimisations futures ;
- dimensions fixes du hero afin de limiter le CLS ;
- mesure du LCP avant et après intégration ;
- validation sur réseau lent ;
- contrôle du coût mémoire ;
- aucun téléchargement de scène inactive.

## 13. Tests

Les tests futurs couvrent :

- le résolveur des sept scènes et la valeur `default` ;
- une météo inconnue ou des données absentes ;
- le jour et la nuit ;
- le changement de ville et de condition ;
- l'échec de chargement d'une image ;
- les changements rapides et les réponses devenues obsolètes ;
- le fallback CSS ;
- le cache dynamique et l'offline ;
- l'unicité du panneau et de la liste ;
- le compteur, la ville active et le focus ;
- les attributs ARIA ;
- l'absence de migration du stockage ;
- la recherche mobile et l'absence de recouvrement ;
- les largeurs de 360 à 1920 pixels ;
- la PWA et la TWA.

## 14. Phases

1. Spécification.
2. Navigation unifiée.
3. Scènes météo.
4. Optimisation desktop.
5. Responsive et accessibilité.
6. Finalisation et publication Web.
7. Mise à jour Android/TWA uniquement après validation Web.

Chaque phase fait l'objet d'une validation technique, visuelle, responsive, fonctionnelle et documentaire avant la suivante.

## 15. Critères d'acceptation

La v1.5.0 peut être publiée uniquement si :

- les sept scènes et le fallback sont disponibles ;
- `clear-day` et `clear-night` sont visuellement distincts ;
- une seule scène est téléchargée pour la condition active ;
- tous les textes restent lisibles ;
- les favoris existants sont conservés sans migration destructive ;
- une seule liste de villes enregistrées est active ;
- le clavier, le focus et les zones tactiles fonctionnent ;
- le zoom et le reflow sont validés ;
- les suggestions de recherche ne sont jamais recouvertes ;
- aucun débordement horizontal de page n'apparaît ;
- le modèle météo et les fournisseurs restent inchangés ;
- la PWA, l'offline, la CSP, la TWA, la validation W3C et la CI sont préservés ;
- les tests automatiques et les contrôles manuels documentés réussissent ;
- la version publique reste 1.4.2 jusqu'à la préparation de la release.
